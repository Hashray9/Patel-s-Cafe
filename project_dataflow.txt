=== SYSTEM ARCHITECTURE OVERVIEW ===
Patel's Cafe PWA is built on a 3-tier client-server architecture:
- CLIENT: React SPA (TypeScript) compiled with Vite, using Tailwind CSS for responsive Neo-Brutalist design.
- API BACKEND: Node/Express server acting as the secure gateway to database resources.
- DATABASE & AUTH: Supabase PostgreSQL database storing cafe configurations and active/historical records, with Supabase Auth handling session authentication.

=== SECURE COMMUNICATION FLOW ===
1. User enters email & password on Login screen.
2. Frontend calls Supabase Auth -> returns active Session token (JWT).
3. Client context retrieves session token dynamically via `supabase.auth.getSession()` on every state change.
4. Client adds JWT in header: `Authorization: Bearer <access_token>` for all API fetches.
5. Express server intercepts incoming requests -> middleware calls `supabase.auth.getUser(<token>)` to verify active session -> attaches user context to `req.user` -> permits request processing or returns 401.

=== DETAILED DATA FLOWS ===
1. Table State & Seating Management Flow:
   - User views Floor Plan -> selects a table.
   - User updates table occupancy status (Seats guests via 'Individual' or 'Group' buttons, or checks out bills).
   - Client sends PUT `/api/tables/:id` with state body (capacity, status, timerStart, currentOrderId, group_id).
   - Server updates database table row in Supabase PostgreSQL -> returns updated table object -> Client context updates React state -> Floor Plan layout re-renders.
   - Seating Group Flow: User clicks "Group" seating -> selects nodes visually on interactive floor map. On save, client sends POST `/api/tables/seat-group` with target `tableIds`.
   - Server creates an empty pending order, generates a `groupId`, updates all selected tables in a transaction block to occupied, and returns status -> client reload state.
   - Custom Table Management: Admins add or delete tables in the Settings panel.
     - Adding table calls POST `/api/tables`. Server validates name uniqueness in PostgreSQL, inserts row, and returns the new table object.
     - Removing table calls DELETE `/api/tables/:id`. Server checks if status is occupied; if so, blocks deletion with 400 Bad Request. Otherwise, deletes table row from database.

2. Order Lifecycle & Billing Flow:
   - Barista/Waiter selects active table -> opens order panel displaying menu items categories.
   - User increments item quantities. Client calculates running subtotal, tax (based on settings taxRate), and total billing figures.
   - Submit Order -> Client sends POST `/api/orders` (or PUT `/api/orders/:id` if updating active order) containing items list, totals, and timestamps.
   - Server updates `orders` database row. If creating a new order on an active table, server updates table status to occupied and associates the order ID.
   - Collapsible Active Orders Queue reads unpaid orders from GET `/api/orders` (filtering by unpaid status).
   - Quantity edits in the order panel update order totals via PUT `/api/orders/:id`. If items list becomes empty, client triggers DELETE `/api/orders/:id` to clean database row, and calls POST `/api/tables/:id/clear` to restore available table layout.
   - Checkout Flow: Clicking "Mark Paid" updates order status to paid via PUT `/api/orders/:id`.
   - Server updates order status in DB. If status is paid, server automatically queries linked table rows, clearing `current_order_id`, `timer_start`, `group_id`, and setting status to `available` for all related tables.

3. Menu Item Administration Flow:
   - User views Menu screen -> filters by category or searches by name query.
   - Client fetches items via GET `/api/menu`.
   - Edit menu item sends PUT `/api/menu/:id` with new price, availability, dietary array, name, and image path.
   - Add menu item sends POST `/api/menu` which inserts a new row and assigns a unique ID.
   - Server writes to `menu_items` table in Supabase PostgreSQL and returns updated/created records to synchronize client view.
   - Delete menu item sends DELETE `/api/menu/:id`.

4. Performance Analytics & Reports Flow:
   - User selects Reports tab.
   - Client calls GET `/api/orders` with query filters: `status=paid` and optional `startDate` / `endDate` timestamps.
   - Since reporting is optimized, server executes fast queried index scans (`idx_orders_status_timestamp` composite index) on orders table in Supabase PostgreSQL, returning historical transactions quickly.
   - Client aggregates orders using React `useMemo` to display total revenue, month-to-date sales, transactions volume, and AOV.
   - Generates sales bar chart coordinates dynamically. If full-screen is requested, enters browser-level fullscreen and requests screen orientation lock.
   - Generates client-side PDF sales report using jsPDF, grouping items by category, calculating category-wise subtotals, and downloading the compiled PDF database layout.

=== ERROR HANDLING STRATEGIES ===
- API Fetch Handlers: Network requests catch errors, log them in the console, and block state updates if responses are invalid.
- Form / Database Validation: Enforces unique table numbers at database level (unique constraints) and price limits/positive counts at client validation level.
- Auth Expired Fallback: Express middleware returns 401 for invalid tokens, which alerts client auth state listeners to redirect the user to the Login screen.
